# KMS 用户认证系统 — 纯自建 JWT

## Goal
为 KMS 添加完整的用户注册/登录/JWT 认证系统，支持多人协作场景。

## 当前状态
- 后端：Go/Gin，MySQL（已有 notes、links 表），已有简单 API Key 中间件（未启用）
- 前端：Next.js，无登录页面，无 token 管理
- 数据库密码：root:root

## 方案

### 后端（`/Users/home/git/kms-server`）

**1. 新增 `users` 表**
```sql
CREATE TABLE users (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  username VARCHAR(64) NOT NULL UNIQUE,
  email VARCHAR(128) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  nickname VARCHAR(64) DEFAULT '',
  role ENUM('admin','member','viewer') DEFAULT 'member',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**2. 新增 `internal/auth/` 包**
- `model.go` — User 结构体、Register/Login 请求体
- `jwt.go` — JWT 签发/解析（access token 2h, refresh token 7d）
- `service.go` — Register（bcrypt 哈希）、Login（验证密码）、GetUser
- `handler.go` — 路由注册
  - `POST /api/auth/register` — 注册
  - `POST /api/auth/login` — 登录，返回 access_token + refresh_token
  - `POST /api/auth/refresh` — 刷新 token
  - `GET  /api/auth/me` — 获取当前用户信息

**3. 新增 `internal/middleware/jwt.go`**
- 解析 Authorization: Bearer <jwt>
- 将 user_id 写入 gin.Context
- 白名单跳过：`/health`, `/api/auth/login`, `/api/auth/register`, `/webhooks/*`

**4. 修改 `cmd/server/main.go`**
- 注册 auth handler 路由
- 将 JWT 中间件替换 APIKeyAuth

**5. 修改 `internal/config/config.go`**
- 新增 `JWT_SECRET` 配置项

### 前端（`/Users/home/kms-web`）

**6. 新增登录/注册页面**
- `src/app/login/page.tsx` — 登录表单
- `src/app/register/page.tsx` — 注册表单

**7. 修改 `src/lib/store.ts`**
- 新增 user 状态、login/logout/register actions
- localStorage 存储 token

**8. 修改 `src/lib/api.ts`**
- 新增 authAPI（login/register/me）
- fetchAPI 自动带 Authorization header
- 401 时自动跳转登录页

**9. 修改 `src/app/layout.tsx`**
- 添加路由守卫：未登录跳转 /login

### 不动的部分
- notes/links 表结构不变
- 现有笔记/搜索/图谱 API 逻辑不变，只是加了鉴权保护
- Gitea 集成不动

## 实现顺序
1. 后端：建表 + auth 包 + JWT 中间件 + 路由注册
2. 前端：登录/注册页 + token 管理 + 路由守卫
3. 联调测试

## 验证
1. 注册新用户 → 201
2. 登录 → 返回 token
3. 带 token 请求 /api/notes → 200
4. 不带 token 请求 → 401
5. 前端未登录 → 跳转 /login
6. 登录后 → 正常使用所有功能

## 风险
- JWT_SECRET 需要配置，否则用默认值不安全
- 前端路由守卫需要处理 token 过期的刷新逻辑
